# 文件重定向

基于 inotifywait 的 Magisk/KSU/APatch 模块，实时监控指定目录，将新增文件按规则自动移动到目标目录，并同步更新媒体库。

## 环境要求

- Magisk / KernelSU / APatch
- Android 8.0+

## 安装

在框架管理器中刷入 zip 安装包即可。升级安装会自动保留现有规则文件和用户配置。

## 规则配置

规则文件位于模块目录下的 `redirect.rules`，可通过模块内置 WebUI 编辑，也可以直接用文本编辑器修改。

每条规则格式如下：

```
源目录|文件名匹配模式|目标目录
```

**说明：**

- 路径支持 `/sdcard/`、`/storage/emulated/0/` 等常见写法，模块内部会自动规范化
- 文件名匹配模式支持 `*`（任意字符）和 `?`（单个字符）通配符，留空或填 `*` 表示匹配全部文件
- `#` 开头的行为注释
- 规则仅匹配源目录本身，不递归匹配子目录

**示例：**

```
# 将相机目录下所有文件移动到备份目录
/sdcard/DCIM/Camera|*|/sdcard/Pictures/Backup

# 只移动 jpg 文件
/sdcard/DCIM/Camera|*.jpg|/sdcard/Pictures/Backup

# 移动特定前缀的文件
/sdcard/DCIM/Camera|example_*|/sdcard/Pictures/Camera
```

规则文件修改后无需重启，模块会自动检测变更并热重载。

## 用户配置

以下参数位于 `service.sh` 顶部，升级时会自动保留：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `STARTUP_SCAN` | `1` | 启动时扫描源目录已有文件，`0` 关闭 |
| `MEDIA_WAIT` | `1` | 文件移动后等待媒体库广播稳定的秒数 |
| `LOG_KEEP_DAYS` | `7` | 日志保留天数，超期自动清空 |
| `DISPATCH_INTERVAL` | `2` | 主循环调度间隔（秒） |

## 日志

日志文件位于模块目录下的 `redirector.log`，可通过 WebUI 查看，也可以直接读取。单文件上限 512KB，超限后自动裁剪保留后半段。

## 模块架构

```
┌─────────────────────────────────────────────────────┐
│                     service.sh                       │
│              主服务 · 单实例守护 · 统筹调度              │
└──────┬────────────────┬───────────────┬─────────────┘
       │                │               │
       ▼                ▼               ▼
┌─────────────┐  ┌─────────────┐  ┌─────────────┐
│ monitor.sh  │  │dispatcher.sh│  │ watchdog.sh │
│ inotifywait │  │ 队列消费·去重 │  │ 30min 心跳  │
│事件监听·防抖  │  │  并发分发    │  │ 崩溃自动拉起 │
└──────┬──────┘  └──────┬──────┘  └─────────────┘
       │                │
       ▼                ▼
  incoming.q  ←─────────┘
  (队列文件)         ▲
       │         retry.q
       └─────►  mv_worker.sh
                同分区 mv
                跨分区 cp + MD5 校验
                    │
                    ▼
              media_fix.sh
              权限修复 · 媒体库同步 · 广播
```

**数据流：**

1. `monitor.sh` 通过 inotifywait 监听源目录，将文件路径写入 `incoming.q`
2. `service.sh` 主循环每 2 秒检测队列，调用 `dispatcher.sh` 消费
3. `dispatcher.sh` 原子交换队列、去重，并发分发给 `mv_worker.sh`
4. `mv_worker.sh` 执行移动，失败写入 `retry.q`，下轮由 `dispatcher.sh` 合并重试
5. 移动成功后调用 `media_fix.sh` 完成权限修复和媒体库更新
6. `redirect.rules` 修改后由 `service.sh` 检测 mtime 变化，自动热重载无需重启

## 实现原理

### 整体架构

模块由五个相互协作的脚本组成，运行于独立进程：

```
service.sh          主服务，单实例守护，统筹调度
├── monitor.sh      监控器，inotifywait 监听目录事件
├── dispatcher.sh   调度器，消费队列，分发给 worker
│   └── mv_worker.sh  搬运工，执行单文件移动和校验
├── watchdog.sh     看门狗，定期检测主服务存活并拉起
└── media_fix.sh    媒体修复，更新媒体库记录和权限
```

### 事件监听

monitor.sh 使用 `inotifywait` 监听 `close_write` 和 `moved_to` 两类事件，分别对应文件写入完成和文件被移入目录这两种场景，确保只处理写入完毕的文件。

同一文件 1 秒内的重复事件会被合并（防抖），避免对同一文件触发多次移动。

监控启动后会立即补扫一次所有源目录，覆盖 inotifywait 建立期间可能漏掉的文件。规则中尚不存在的目录会由独立的轮询子进程持续等待，目录出现后自动加入监控并补扫。

### 队列与调度

文件路径由 monitor.sh 写入 `incoming.q` 队列，dispatcher.sh 定期消费。调度时先将队列文件原子交换出来，合并上轮失败的重试条目，再经 `sort -u` 去重，最后按规则匹配后并发分发给 mv_worker.sh 处理。

并发 worker 数量根据设备 CPU 核心数自动计算，下限 2，上限 16。dispatcher.sh 使用 `mkdir` 原子锁防止启动扫描与主循环调度并发执行。

### 文件移动

mv_worker.sh 通过比较源文件与目标目录的设备号判断是否跨分区：

- **同分区**：直接 `mv`，原子操作，无数据安全风险
- **跨分区**：`cp` 后对比文件大小和 MD5，校验通过后再删除源文件，失败自动重试一次

重名文件通过在文件名末尾追加序号（最多 99 次）解决冲突，仍然冲突则跳过并记录日志。

### 媒体库同步

文件移动完成后，media_fix.sh 依次执行三步：修正文件权限（`1023:1023 / 0664`，与 Android 媒体文件标准一致）、通过 `content` 命令删除媒体库中的旧路径记录（同时处理 `/sdcard/` 和 `/storage/emulated/0/` 两种路径形式）、广播 `MEDIA_SCANNER_SCAN_FILE` 通知系统扫描新路径。

### 可靠性设计

- **单实例保护**：主服务通过 `mkdir` 原子锁 + PID 文件双重校验防止重复启动，校验时读取 `/proc/PID/cmdline` 验证进程身份，防止 PID 复用导致误判
- **看门狗**：每 30 分钟检测主服务存活，异常停止时自动拉起，模块被框架禁用时静默退出
- **OOM 保护**：主服务和看门狗启动时将 `oom_score_adj` 设为 `-1000`，防止被系统内存回收机制杀掉
- **Shell 兼容性**：全部脚本遵循 POSIX Shell 标准，兼容 mksh 和 busybox ash，不使用任何非标准扩展
